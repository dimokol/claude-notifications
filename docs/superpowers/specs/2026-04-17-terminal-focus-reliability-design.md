# Terminal Focus Reliability — Design

**Date:** 2026-04-17
**Scope:** Track 1 of the cross-build compatibility effort. Addresses the bug where clicking an OS banner (or in-window toast) opens the correct VS Code window but lands on the wrong terminal tab, particularly when multiple Claude terminals share a display name.
**Out of scope:** Phantom notification firing after a signal has already been dismissed (Track 2). Signal-layer changes in `hook.js` or `lib/signals.js`. Any modification to the claim/dedup protocol.

## Problem

Log evidence captured against Claude Notifications v3.1.2 running against the native (compiled) Claude CLI build shows that PID-based matching in `focusMatchingTerminal` succeeds in every click-to-focus case. No `Name match` or `Fallback` path is ever hit. Despite this, users perceive the wrong terminal being focused in two situations:

1. When two tabs share a display name (e.g. two `2.1.112` tabs running native Claude), the log line `Active terminal after switch: "2.1.112"` is ambiguous — we cannot tell from the log which tab VS Code actually activated.
2. When the user switches to another app, clicks the OS banner, and returns to VS Code while a *different* Claude terminal was previously active, the new tab fails to receive focus despite `terminal.show()` being called on the correct reference.

The leading hypothesis is that `showTerminal` calls `workbench.action.terminal.focus` *before* `terminal.show()`. That panel-focus command only focuses the panel (preserving whichever terminal was last-active); the subsequent `show()` then races with the panel-open transition and occasionally does not override the active tab. `terminal.show()` already reveals the panel and focuses the specific terminal, so the preceding command is both redundant and a likely source of the race.

The name-based fallback at `extension.js:338-345` was considered for a version-aware update, but was deliberately left alone: with every Claude terminal sharing a uniform display name (`"node"`, `"claude"`, or a version string), a substring match cannot pick the *correct* terminal among several — it just picks the first. PID matching, which is build-agnostic and handles every case in the captured logs, remains the only disambiguating path. Changing the name fallback would not improve correctness.

## Design

Two focused changes, both inside `extension.js`. No changes to `hook.js`, `lib/signals.js`, or the signal-file protocol.

### 1. Simplify `showTerminal` (extension.js:357-364)

Drop the preceding `workbench.action.terminal.focus` command. The `terminal.show()` API already reveals the panel and focuses the specific terminal reference. Removing the redundant command eliminates the race between panel-open and show-target.

**Before:**
```js
async function showTerminal(terminal, log) {
  await vscode.commands.executeCommand('workbench.action.terminal.focus');
  terminal.show();
  setTimeout(() => {
    const active = vscode.window.activeTerminal;
    log.appendLine(`Active terminal after switch: "${active?.name || 'none'}"`);
  }, 300);
}
```

**After:**
```js
async function showTerminal(terminal, log) {
  terminal.show();
  setTimeout(async () => {
    const active = vscode.window.activeTerminal;
    const label = await describeTerminal(active);
    log.appendLine(`Active terminal after switch: ${label}`);
  }, 300);
}
```

### 2. Disambiguating diagnostics in `focusMatchingTerminal`

When two tabs share a name, the current log cannot distinguish them. The fix is to include the terminal's index in `vscode.window.terminals` and its process ID in every log line that references a terminal.

Introduce a small `describeTerminal(terminal, index)` helper that formats as `[<index>]"<name>"(pid=<pid>)`, reused by the open-terminals list, the match line, and the active-after-switch line. Because `terminal.processId` is a promise, the helper awaits it; when it cannot be resolved, format as `[<index>]"<name>"(pid=?)`.

Log shape after the change:

```
Open terminals (4): [0]"node"(pid=53306), [1]"2.1.112"(pid=75623), [2]"2.1.112"(pid=89403), [3]"zsh"(pid=91111)
PID match: [1]"2.1.112"(pid=75623)
Active terminal after switch: [1]"2.1.112"(pid=75623)
```

An index mismatch between the match line and the active-after-switch line becomes visible evidence that `show()` did not take effect, turning a previously-invisible failure into a diagnosable one.

## Files Modified

- `extension.js` — two changes described above, plus a small `describeTerminal` helper co-located with `focusMatchingTerminal`. Estimated net diff: ~20 lines.

No changes to `dist/extension.js` by hand — it is regenerated by `esbuild.js`.

## Testing Plan

Manual verification because this code path is tightly coupled to VS Code window and terminal state, which is not readily unit-testable:

1. Open two Claude terminals in the same VS Code window, both running native Claude (both tabs display as `2.1.112`).
2. In terminal A, trigger a notification; click in-window "Focus Terminal" while already on terminal B. Verify: focus lands on A, and the output channel shows matching `[<index>]` values between `PID match` and `Active terminal after switch`.
3. Trigger a notification in terminal A. Switch to another app. Click the OS banner. Verify: VS Code returns with terminal A focused, and log indices match.
4. Repeat step 3 using the Homebrew Claude build in one terminal and native in another, to confirm PID match still works across mixed builds.

Regression surface: any existing flow that relied on the panel-focus command having run first. None known, but worth watching for "terminal tab visually switches but cursor does not land there" reports — unlikely, since `show()`'s default is `preserveFocus: false`.

## Success Criteria

- Output channel log always contains enough information to determine which specific terminal tab was matched and which one actually became active.
- Click-to-focus from another window reliably lands on the correct terminal when two tabs share a display name.
